    # Getting Started: Create a new repo using the ArcGIS Pro Python environment
     
    This how-to guide walks geospatial data scientists through creating a **new GitHub repository** and working locally using the **Python environment that comes with ArcGIS Pro** (so you can use `arcpy` and related Esri Python packages).
     
    ## Audience and assumptions
     
    **Audience:** Geospatial data scientists who write Python scripts/notebooks for ArcGIS Pro workflows.
     
    **You should already have:**
     
    - ArcGIS Pro installed (Windows).
    - A GitHub account.
    - Git installed on your machine.
    - Permission to create repositories in your GitHub org (if applicable).
     
    ## Why use the ArcGIS Pro Python environment?
     
    - It is the supported way to use Esri tooling like `arcpy`.
    - It reduces “works on my machine” issues compared to ad-hoc Python installs.
     
    **Important constraint:** the ArcGIS Pro environment is Windows-based and tied to the ArcGIS Pro install. Reproducing it on Linux CI runners is usually not feasible. Plan your repo accordingly (see “Execution & reproducibility notes” below).
     
    ## Overview (what you will do)
     
    1. Create a new GitHub repository.
    2. Clone it locally.
    3. Create a project folder structure.
    4. Create a Python entry point and verify you can import `arcpy`.
    5. Capture environment metadata for reproducibility.
    6. Commit and push.
     
    ## Step 1 — Create a new GitHub repository
     
    In GitHub:
     
    1. Click **New repository**.
    2. Name it using a clear, task-oriented name (examples: `wetland-mapping`, `lidar-qaqc`, `levee-inspection-tools`).
    3. Choose **Private** vs **Public** per your organization policy.
    4. Initialize with:
       - `README.md`
       - `.gitignore` (choose **Python** as a starting point)
       - License (if public; follow your policy)
     
    **Recommended README sections (minimum):**
    - Purpose
    - How to run (ArcGIS Pro Python)
    - Inputs/outputs
    - Data sources (and whether data is committed or external)
     
    ## Step 2 — Clone the repo locally
     
    Choose a working location (example: `C:\git\`), then clone:
     
    - Open a terminal (PowerShell is fine).
    - Run:
      - `git clone <REPO_URL>`
      - `cd <REPO_FOLDER>`
     
    Verify:
    - `git status` works and you are on the default branch.
     
    ## Step 3 — Create a simple, predictable project structure
     
    Create these folders (adjust as needed, but keep it consistent across projects):
     
    - `src/` — reusable Python modules
    - `scripts/` — runnable scripts (entry points)
    - `notebooks/` — exploratory work (optional)
    - `data/` — small local data only (see note below)
    - `outputs/` — generated outputs (usually not committed)
    - `docs/` — documentation assets (optional)
     
    Data handling note:
    - Do **not** commit large rasters/geodatabases to Git unless you have an explicit pattern (Git LFS, external storage, etc.).
    - Prefer keeping authoritative data in approved storage and documenting access.
     
    ## Step 4 — Use the ArcGIS Pro Python environment (and verify `arcpy`)
     
    Goal: run Python using the same environment ArcGIS Pro uses.
     
    ### Option A (recommended) — Use ArcGIS Pro’s Python Command Prompt
     
    1. Open the **ArcGIS Pro Python Command Prompt** from the Start Menu.
    2. In that prompt, navigate to your repo folder:
       - `cd C:\git\<REPO_FOLDER>`
    3. Confirm you are using the ArcGIS Pro Python:
       - `python --version`
       - `python -c "import sys; print(sys.executable)"`
     
    Create a smoke-test script:
     
    - Create `scripts/smoke_test_arcpy.py`
    - Put this minimal code inside (keep it short and deterministic):
      - Import `arcpy`
      - Print `arcpy.GetInstallInfo()` (or a small subset)
      - Print the Python executable path
     
    Run it:
    - `python scripts\smoke_test_arcpy.py`
     
    Verification:
    - The script runs without import errors.
     
    ### Option B — Use the conda env that ArcGIS Pro ships (when applicable)
     
    Some ArcGIS Pro installs expose conda environments (via `conda.bat`) and environment names like `arcgispro-py3`.
     
    If your org supports this workflow, document:
    - the environment name,
    - how to activate it,
    - and how to run scripts consistently.
     
    If you are unsure which approach your team standardizes on, use Option A and document exactly what you did.
     
    ## Step 5 — Capture environment metadata for reproducibility
     
    Because ArcGIS Pro’s environment is not a typical pip-only environment, treat environment capture as “record what you have”, not “recreate everywhere”.
     
    Add an `env/` folder and capture:
     
    - `env/arcgispro_python_executable.txt`
      - contains the output of: `python -c "import sys; print(sys.executable)"`
     
    - `env/python_version.txt`
      - contains the output of: `python --version`
     
    - `env/pip_freeze.txt` (best effort)
      - contains the output of: `python -m pip freeze`
     
    - `env/arcpy_install_info.txt`
      - contains the output of a small script calling `arcpy.GetInstallInfo()`
     
    Also record:
    - ArcGIS Pro version (exact, as shown in ArcGIS Pro “About” dialog)
    - Windows version (optional but helpful)
     
    ## Step 6 — Add a minimal “first workflow” script
     
    Create a small script that proves the repo can do one meaningful GIS task in a deterministic way.
     
    Suggestions (choose one):
    - List feature classes in a file geodatabase path (parameterized).
    - Create a scratch geodatabase in `outputs/` and write a tiny feature class from known coordinates.
    - Validate that a required extension/license is available (and fail with a clear message if not).
     
    Keep inputs explicit and avoid relying on local state that another analyst can’t reproduce.
     
    ## Step 7 — Commit and push
     
    1. `git status`
    2. `git add -A`
    3. `git commit -m "Initialize ArcGIS Pro Python repo (arcpy smoke test and env metadata)"`
    4. `git push`
     
    Verification:
    - Repo on GitHub shows your structure, scripts, and `env/` metadata files.
     
    ## Execution & reproducibility notes (important)
     
    - **CI runners:** Most GitHub-hosted runners will not have ArcGIS Pro installed. If you add CI, either:
      - run only linting/unit tests that do not require `arcpy`, or
      - use a self-hosted runner with ArcGIS Pro installed (if policy allows).
    - **Environment recreation:** Treat ArcGIS Pro as the source of truth; document versions and capture metadata so others can align their installs.
    - **Data provenance:** Document where data comes from and how to obtain it; don’t rely on undocumented shared drives or local paths.
     
    ## Troubleshooting
     
    - `ImportError: No module named arcpy`
      - You are not running the ArcGIS Pro Python environment. Re-run using the ArcGIS Pro Python Command Prompt and confirm `sys.executable`.
     
    - Script works in ArcGIS Pro GUI but not in terminal
      - Ensure you launched the correct prompt and are not mixing Python installs.
     
    - Permissions or network drive paths fail
      - Prefer local working paths for intermediate outputs and parameterize input paths.
     
    ## Definition of done (checklist)
     
    - [ ] New GitHub repo created with README and Python `.gitignore`
    - [ ] Repo cloned locally; `git status` clean
    - [ ] Project structure created (`src/`, `scripts/`, `outputs/`, etc.)
    - [ ] `scripts/smoke_test_arcpy.py` runs successfully
    - [ ] `env/` metadata captured (python executable, version, pip freeze, arcpy install info)
    - [ ] At least one minimal, deterministic GIS workflow script added
    - [ ] Changes committed and pushed

    # Create an ArcGIS Pro Python Repo (ArcGIS Pro 3.x / `arcgispro-py3`)
    
    ## Introduction
    
    This chapter shows how to start a new **geospatial Python** repository that uses the **highly customized Python environment shipped with ESRI ArcGIS Pro 3.x**.
    
    Key ideas:
    
    - ArcGIS Pro ships with a conda-based environment (commonly named `arcgispro-py3`) that is configured to work with ArcGIS Pro and `arcpy`.
    - For project work, we **clone** `arcgispro-py3` into a **per-repository conda environment** (one environment per repo).
    - We use **VS Code** as the IDE and the **VS Code Git interface** for day-to-day Git operations.
    - We avoid fragile “roll-your-own activation” tricks and instead leverage ESRI’s supported activation scripts / prompts.
    
    Canonical ESRI documentation:
    - Installing Python for ArcGIS Pro (and environment management): https://pro.arcgis.com/en/pro-app/latest/arcpy/get-started/installing-python-for-arcgis-pro.htm
    
    USACE note on software installs:
    - If you do not have ArcGIS Pro 3.x, VS Code, or Git installed, follow the **approved USACE employee workflows** documented elsewhere in this user guide (see related chapters in the Git Repos / Getting Started sections). Do not invent an ad-hoc install path.
    
    ## Key steps (overview)
    
    1. Create a new repository in GitHub (from an approved template repo, if available).
    2. Clone the new repository to your computer using the VS Code Git interface.
    3. Use ArcGIS Pro’s Python tooling to identify and verify `arcgispro-py3`.
    4. Clone `arcgispro-py3` into a per-repo environment (example name: `.conda-arcgis`).
    5. Configure VS Code to use the cloned environment interpreter.
    6. Verify `arcpy` imports and a minimal GIS workflow runs.

    5. When prompted, open the cloned repository.

    ## Step 3 — Verify you can access the ArcGIS Pro Python environment (`arcgispro-py3`)
    
    ArcGIS Pro’s Python environment is not a “generic” conda install; it is customized to support `arcpy` and ArcGIS Pro integration. Use ESRI’s supported entry points.

    name=env/capture_env_metadata.py
    
    from __future__ import annotations
    
    import argparse
    import os
    import subprocess
    import sys
    from datetime import datetime
    from pathlib import Path
    
    
    def run_cmd(cmd: list[str]) -> tuple[int, str]:
        """
        Run a command and capture stdout+stderr as text.
        Returns (returncode, combined_output).
        """
        proc = subprocess.run(
            cmd,
            stdout=subprocess.PIPE,
            stderr=subprocess.STDOUT,
            text=True,
            shell=False,
        )
        return proc.returncode, proc.stdout
    
    
    def write_text(path: Path, text: str) -> None:
        path.write_text(text, encoding="utf-8", newline="\n")
    
    
    def main() -> int:
        parser = argparse.ArgumentParser(
            description="Capture lightweight environment metadata for ArcGIS Pro Python repos."
        )
        parser.add_argument(
            "--pip-freeze",
            action="store_true",
            help="Also write env/pip_freeze.txt (often large).",
        )
        args = parser.parse_args()
    
        out_dir = Path(__file__).resolve().parent
        out_dir.mkdir(parents=True, exist_ok=True)
    
        timestamp = datetime.now().isoformat(timespec="seconds")
    
        # Basic Python facts (always)
        write_text(out_dir / "timestamp.txt", f"{timestamp}\n")
        write_text(out_dir / "python_executable.txt", f"{sys.executable}\n")
        write_text(out_dir / "python_version.txt", f"{sys.version}\n")
    
        # arcpy version (best effort)
        try:
            import arcpy  # type: ignore
    
            write_text(
                out_dir / "arcpy_version.txt",
                f"{getattr(arcpy, '__version__', 'UNKNOWN')}\n",
            )
        except Exception as e:
            write_text(
                out_dir / "arcpy_version.txt",
                "FAILED to import arcpy. Activate .conda-arcgis and try again.\n"
                f"Error: {repr(e)}\n",
            )
    
        # Conda metadata (best effort)
        # These will only work if the shell session was initialized via proenv.bat.
        conda_cmds: list[tuple[str, list[str]]] = [
            ("conda_version.txt", ["conda", "--version"]),
            ("conda_info.txt", ["conda", "info"]),
            ("conda_env_list.txt", ["conda", "env", "list"]),
        ]
    
        for filename, cmd in conda_cmds:
            rc, out = run_cmd(cmd)
            header = f"Command: {' '.join(cmd)}\nReturn code: {rc}\n\n"
            write_text(out_dir / filename, header + out)
    
        # pip freeze is optional and flag-gated
        if args.pip_freeze:
            rc, out = run_cmd([sys.executable, "-m", "pip", "freeze"])
            header = (
                f"Command: {sys.executable} -m pip freeze\n"
                f"Return code: {rc}\n\n"
            )
            write_text(out_dir / "pip_freeze.txt", header + out)
    
        # Friendly summary
        wrote = [
            "timestamp.txt",
            "python_executable.txt",
            "python_version.txt",
            "arcpy_version.txt",
            "conda_version.txt",
            "conda_info.txt",
            "conda_env_list.txt",
        ]
        if args.pip_freeze:
            wrote.append("pip_freeze.txt")
    
        summary = "Wrote:\n" + "\n".join([f"- env/{f}" for f in wrote]) + "\n"
        print(summary)
        return 0
    
    
    if __name__ == "__main__":
        raise SystemExit(main())
