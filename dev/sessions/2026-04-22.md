## Options — Python library stack for PDF → semantic chunks + metadata

Ranked highest-confidence first, based on: provenance richness, ecosystem maturity, platform-agnosticism, and auditability.

---

### Option 1 ⭐ `unstructured` + LangChain `SemanticChunker`
**What it is:** `unstructured` (Unstructured.io) handles PDF parsing and emits richly typed document elements (NarrativeText, Title, Table, Header, Footer, etc.) with per-element metadata. LangChain's `SemanticChunker` groups sentences by embedding similarity, producing semantically coherent chunks.

| | |
|---|---|
| **PDF parsing metadata** | Page number, element type, filename, file path, last-modified date, coordinates/bounding box, languages detected |
| **Chunking strategy** | Embedding-based semantic similarity (cosine distance threshold) |
| **Provenance depth** | ✅ Best-in-class — element-level origin tracking survives chunking |
| **Ecosystem maturity** | High — both libraries are widely documented and actively maintained |
| **Tradeoffs** | `unstructured` has significant optional dependencies (for high-fidelity PDF: `pdfminer`, `pikepdf`, optionally `tesseract` for scanned docs); LangChain is a large dependency but `langchain-text-splitters` can be installed standalone |
| **Output flexibility** | JSONL for Gemini; Parquet for Foundry — both straightforward |
| **Risk** | LangChain API surface changes frequently; pin versions |

---

### Option 2 `docling` (IBM) + `chonkie`
**What it is:** IBM's `docling` is a newer PDF parser that preserves document structure (headings, sections, tables, lists) and emits a rich DoclingDocument schema. `chonkie` is a lightweight, purpose-built chunking library with a `SemanticChunker`.

| | |
|---|---|
| **PDF parsing metadata** | Section headings, page number, bounding boxes, table/figure labels, document hierarchy |
| **Chunking strategy** | Semantic (embedding-based), token-aware |
| **Provenance depth** | ✅ Strong — structural hierarchy (section path) is preserved per chunk |
| **Ecosystem maturity** | Medium — both are actively developed but newer; fewer production case studies than Option 1 |
| **Tradeoffs** | `docling` excels at structured/formatted PDFs (reports, regulations); may underperform on scanned or unstructured PDFs; `chonkie` has less community documentation |
| **Output flexibility** | Same — JSONL + Parquet |
| **Risk** | Smaller community = fewer Stack Overflow answers; API less stable |

---

### Option 3 `PyMuPDF` + custom semantic chunker (`sentence-transformers`)
**What it is:** `PyMuPDF` (fitz) is a fast, low-level PDF parser. You write a custom semantic chunker using `sentence-transformers` cosine similarity — sliding window grouping sentences until similarity drops below threshold.

| | |
|---|---|
| **PDF parsing metadata** | Page number, PDF metadata dict (title, author, subject, creation date, producer), block coordinates |
| **Chunking strategy** | Custom cosine-similarity sentence grouper — fully transparent |
| **Provenance depth** | ✅ You control every metadata field — maximum auditability |
| **Ecosystem maturity** | `PyMuPDF` is very mature; `sentence-transformers` is mature; the custom chunker is your code |
| **Tradeoffs** | Most code to write and maintain; custom logic must be tested and documented; no abstraction layer handles edge cases automatically |
| **Output flexibility** | Same |
| **Risk** | Maintenance burden; edge cases (multi-column PDFs, footnotes) require custom handling |

---

### Option 4 `unstructured` end-to-end (single library)
**What it is:** `unstructured` alone handles both parsing and chunking (`chunk_by_title()` groups elements under their nearest heading; `chunk_elements()` does fixed-size chunking with overlap). No separate chunking library needed.

| | |
|---|---|
| **PDF parsing metadata** | Same as Option 1 |
| **Chunking strategy** | Title-based (structural) or fixed-size with overlap — **not embedding-based semantic chunking** |
| **Provenance depth** | ✅ Same rich element metadata as Option 1 |
| **Tradeoffs** | `chunk_by_title` is structural, not semantic — chunks follow document headings, not embedding similarity. This may be good enough for well-structured reports, but doesn't satisfy "semantic chunking" in the embedding sense. |
| **Output flexibility** | Same |
| **Risk** | If documents lack consistent heading structure (e.g., scanned PDFs, legacy reports), chunk quality degrades |

---

### Option 5 `pdfplumber` + recursive character splitter (LangChain)
**What it is:** `pdfplumber` extracts text and page metadata; LangChain's `RecursiveCharacterTextSplitter` chunks by character count with overlap.

| | |
|---|---|
| **PDF parsing metadata** | Page number, file path, bounding boxes |
| **Chunking strategy** | Fixed-size with overlap — **not semantic** |
| **Provenance depth** | ⚠️ Adequate but not best-in-class — no element typing, no section heading tracking |
| **Tradeoffs** | Simple, fast, minimal dependencies — but fixed-size chunking splits mid-sentence, mid-paragraph; violates the semantic coherence requirement |
| **Output flexibility** | Same |
| **Risk** | Does not meet your stated goal of "semantic chunking best practice" — not recommended given the audit trail requirements |

---

## My recommendation

**Option 1** (`unstructured` + LangChain `SemanticChunker`) for the initial version, with a clear note that `docling` (Option 2) should be evaluated as it matures. Rationale:

- Richest per-element metadata out-of-box (critical for your audit trail requirement)
- `SemanticChunker` is well-documented with a stable API when version-pinned
