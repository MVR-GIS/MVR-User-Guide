24  Initiate a reproducible Threads workflow

24.1 Objective

Initiate a reproducible workflow for capturing a Threads chat session history and saving outputs in a reviewable, auditable way inside a Foundry enclave (assume offline / no-internet execution).

This chapter focuses on workflow and structure, not on account onboarding or Foundry project setup.

24.2 Prerequisites

Complete these first:

  • fnd-getting-started.qmd
    • login
    • create a Project
    • tagging for discovery
    • groups/permissions
    • any required platform conventions
  • fnd-load-repo.qmd
    • import your GitHub analysis repository into a Foundry Code Repository using ZIP import

You should also already have a GitHub analysis repo created from:

  • repo-create-generic-python-repo.qmd

24.3 What is a “reproducible Threads session” in this context?

A reproducible Threads session should leave an audit trail that includes:

  • who/what: analyst, date/time window, project context
  • inputs: the prompt(s), any system/context constraints, and referenced datasets
  • configuration: model/tool settings as allowed by your environment
  • outputs: generated text, extracted entities/tables, files produced
  • provenance: dataset RIDs/paths (or equivalent identifiers), code version/commit, and run identifier

Your goal is that a reviewer can answer:

  • “What question was asked?”
  • “What data and code were used?”
  • “What did the assistant produce?”
  • “Where are the saved artifacts?”

24.4 Key steps (overview)

  1. Ensure your Foundry Code Repo is imported and ready.
  2. Create a standard session folder for the Threads run.
  3. Capture session metadata (who/what/why, dataset references, config).
  4. Capture the session transcript (prompts + responses) in a stable format.
  5. Save outputs (and any derived artifacts) with deterministic naming.
  6. Commit changes to the Code Repo to preserve an audit trail (use Git LFS if needed for large artifacts).

24.5 Recommended repository structure for Threads sessions

Use a stable location inside the repo for session artifacts. Recommended pattern (create if missing):

  • threads/
    • sessions/
      • <YYYY-MM-DD>_<short-topic-slug>/
        • README.md
        • metadata.yml
        • transcript.md
        • outputs/

Notes:

  • Keep “session history” in plain text formats (.md, .yml, .json) to maximize portability and diffability.
  • Avoid proprietary binary formats for primary records.
  • Default: store all session artifacts in the repository so the repo is the durable, auditable system-of-record.
  • Large artifacts:
    • Prefer smaller, reviewable derivatives (summaries, samples, compressed exports) when practical.
    • If you must keep large files, Git LFS is acceptable—use it intentionally and keep file naming and metadata clear.

24.6 Step 1 — Create a new session folder

Create a new folder under:

  • threads/sessions/

Naming convention:

  • YYYY-MM-DD_short-topic-slug

Example:

  • 2026-05-03_real-estate-permits-qaqc

24.7 Step 2 — Record metadata first (before you chat)

Create metadata.yml with, at minimum:

  • session name
  • date
  • author
  • purpose / question
  • datasets referenced (identifiers, not copies)
  • constraints (offline/no-internet, sensitive handling notes)
  • code provenance (commit SHA if available)

Recommendation:

  • include a section that explicitly states what is out of scope or what was not done (helps reviewers).

24.8 Step 3 — Run the Threads session

Conduct the Threads chat session following your organization’s policy on:

  • what data can be pasted into chat
  • how sensitive data is handled
  • what model/tooling is approved

During the session, keep a scratchpad of:

  • each prompt you send
  • each key response you receive
  • any action items or decisions

24.9 Step 4 — Save the transcript (stable + reviewable)

Create transcript.md and capture:

  • the prompts (verbatim)
  • the responses (verbatim)

If your platform provides an export function, prefer that over manual copy/paste, but ensure:

  • the format is plain text / markdown
  • timestamps (if present) are consistent and not misleading
  • you remove/redact sensitive data per policy

24.10 Step 5 — Save outputs (and keep them traceable)

Put any derived artifacts in outputs/, such as:

  • a cleaned “final answer” version of the response
  • extracted tables as CSV
  • checklists or SOP steps as markdown

Naming guidance:

  • keep names deterministic and descriptive
  • avoid spaces; use hyphens

Examples:

  • outputs/final-summary.md
  • outputs/extracted-entities.csv
  • outputs/recommended-workflow-checklist.md

24.11 Step 6 — Preserve an audit trail

Minimum viable audit trail:

  • metadata.yml exists and references:
    • datasets used (identifiers)
    • where outputs are stored
    • the repo version / commit (or a note that this is not available)
  • transcript.md is present
  • outputs are saved under outputs/

Preferred:

  • commit the complete session folder to the Foundry Code Repo with a clear commit message
  • if you use Git LFS for large files, ensure the LFS pointers are committed and the files are actually present in the remote LFS store (so others can retrieve them).

24.12 Offline / no-internet constraints

Design your workflow so it does not require:

  • downloading packages at run time
  • calling external APIs
  • embedding links as “required steps” unless internal and accessible

If you reference external documentation, treat it as “background reading” rather than a dependency for completing the workflow.

24.13 Troubleshooting

I can’t import the repo ZIP into Foundry

  • Re-check fnd-load-repo.qmd
  • Confirm you have permissions to create or modify a Code Repository in the target Project
  • Confirm you are using an approved transfer path for moving the ZIP into the enclave

I’m not sure what I’m allowed to paste into Threads

  • Stop and confirm policy with your platform/data governance point of contact
  • Prefer referencing datasets by identifier rather than pasting raw records

Outputs are too large to store in the repo

Default expectation: store artifacts in-repo (the repo is the durable audit record).

Practical options:

  • Store a smaller derivative (summary, sample, or compressed version) in outputs/.
  • If you need the full artifact, use Git LFS for the large file(s).
  • Record what you did in metadata.yml so a reviewer understands what is complete vs summarized.

24.14 Definition of done

You are done when:

  • a new session folder exists under threads/sessions/
  • metadata.yml exists and includes:
    • purpose/question
    • dataset identifiers (if any)
    • constraints (offline/no-internet)
    • where outputs were saved
  • transcript.md contains a complete prompt/response record (redacted as needed)
  • outputs are saved and referenced
  • the work is preserved in a way a reviewer can retrieve later (commit and/or published artifacts)
